调整打印方式

我们并不总是希望将所有的日志都打印到控制台上,而是希望将其放在某个文件中方便我们保存.这涉及到日志系统中级工的打印处理器handler,常用的handler包括以下三类

  • ConsoleHandler - 将日志通过System. err打印到控制台，现在默认就是使用的这个。
  • FileHandler - 将日志直接写入到指定的文件中。
  • SocketHandler - 将日志利用Socket通过网络发送到另一个主机。

日志的继承关系

我们刚才提到了所有的日志均继承自rootLogger，但不同的日志之间也有类似的继承关系。所有的日志会按照包的名称进行继承

所有的日志对象在创建时会继承父Logger的配置,所以我们才能够在上面提到的配置文件中为所有的日志对象配置统一的属性

SLF4j

之前提到现在主要使用SLF4j与log4j2搭建项目的日志,我们知道了后者是一个日志框架,那么前者被,前者被称为日志门面(Logging Facade),简单的理解就是一套与日志相关的api,其他的日志框架通过实现这套api来完善日志功能而SLF4j(Simple Logging Facade for Java)则是目前使用最广泛的一套API,许多的主流框架如Spring,MyBaits等都使用了它,所以我们可以通过它来方便的获取这些框架的日志

此外，得益于统一的接口，我们甚至可以无缝的在不同的日志框架间切换

我们之前引入log4j时分别引入了-api和-core，其实前者是log4j自带的日志门面，我们现在已经不需要了转而改为

剩下的还需要提一嘴的就只有对于SLF4j来说日志主要分为包括TRACE、DEBUG、INFO、WARN和ERROR，和log4j略有不同

断言工具

接下来我们要学习断言,即下判断.上面我们的测试程序无论怎么写只要程序正常运行就会显示测试成功,哪怕实际的运行结果不符合我们的期望.为了解决这一问题,我们需要使用断言来判断程序是否运行成功

JUnit 5 断言（Assertions）常用方法汇总如下：

方法 说明 示例
assertEquals(expected, actual) 断言 预期值 和 实际值 相等 assertEquals(10, sum(5, 5));
assertNotEquals(unexpected, actual) 断言两个值不相等 assertNotEquals(10, sum(4, 5));
assertTrue(condition) 断言条件为 true assertTrue(value > 0);
assertFalse(condition) 断言条件为 false assertFalse(list.isEmpty());
assertNull(object) 断言对象为 null assertNull(user.getEmail());
assertNotNull(object) 断言对象不为 null assertNotNull(user.getName());
assertSame(expected, actual) 断言两个对象是 同一个引用 assertSame(obj1, obj2);
assertNotSame(unexpected, actual) 断言两个对象 不是同一个引用 assertNotSame(new String("hello"), new String("hello"));
assertArrayEquals(expected, actual) 断言两个数组相等 assertArrayEquals(new int[]{1,2,3}, new int[]{1,2,3});
assertThrows(Exception.class, executable) 断言代码抛出指定异常 assertThrows(IllegalArgumentException.class, () -> method(null));
assertDoesNotThrow(executable) 断言代码 不会 抛出异常 assertDoesNotThrow(() -> method("valid"));
assertTimeout(duration, executable) 断言代码在指定时间内执行完毕 assertTimeout(Duration.ofMillis(500), () -> slowMethod());
fail(message) 强制测试失败 fail("This test should not be reached!");

上面的所有断言方法都可以在末尾添加message

测试中的顺序控制

我们的部分测试需要考虑到顺序的影响,在默认情况下测试将会按照类中方法的顺序进行,但可以通过注解改变顺序

此外还可以使用随机化的测试顺序

注解@TestMethodOrder可选的选项包括

类型 作用
MethodOrderer.DisplayName 根据测试名称按照字典顺序进行排序
MethodOrderer.MethodName 根据方法名称按照字典顺序排序
MethodOrderer.OrderAnnotatio 根据Order注解的顺序排序
MethodOrderer.Random 随机进行测试

有些时候,我们需要在测试前做一些准备,或者在测试完成之后执行一些收尾工作

有@BeforeAll注解的方法(必须是static方法)将会在该类的测试开始之前进行执行

包含@BeforeEach注解的方法将会在每个测试用例开始之前执行

有@AfterAll注解的方法将会在该类的所有测试用例结束后执行.还有一个@AfterEach注解就不演示了,会在每个测试用例执行完毕后执行

生命周期

在测试中,默认情况下每一个测试用例都将会创建一个新的对象再去执行,在某些耗费大量性能的测试中,这是不可接受的,我们可以通过注解调整
